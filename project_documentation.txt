ПРОЕКТНАЯ ДОКУМЕНТАЦИЯ

Информационная система учета заказов и расчета материальной стоимости продукции
Вариант 4
Организация: ООО "Два сеньора"
База данных: V4


1. НАЗНАЧЕНИЕ ПРОЕКТА

Проект разработан для автоматизации учета заказов покупателей, готовой продукции, материалов и спецификаций производства. Система позволяет хранить сведения о контрагентах, учитывать состав продукции, фиксировать заказ покупателя и рассчитывать полную материальную стоимость заказа.

В рамках варианта 4 в систему внесены данные по организации ООО "Два сеньора", заказчику ООО "Кинотеатр Квант", продукции категории "пицца" и материалам, входящим в спецификацию пиццы "Пеперони" 33см.

Проект состоит из пяти модулей:

Модуль 1 - ER-диаграмма базы данных.
Модуль 2 - создание структуры базы данных и загрузка данных.
Модуль 3 - SQL-запрос расчета полной материальной стоимости заказа.
Модуль 4 - приложение авторизации пользователей.
Модуль 5 - проектная документация.

В базу данных загружаются заказчики, материалы, продукция, спецификация продукции и заказ покупателя.

3. СТРУКТУРА БАЗЫ ДАННЫХ

База данных приведена к простой нормализованной структуре. Данные разделены по сущностям, чтобы не дублировать информацию о заказчиках, товарах, материалах и составе продукции.

Таблица customers хранит сведения о контрагентах:
- id - код контрагента, первичный ключ;
- name - наименование организации;
- inn - ИНН;
- address - адрес;
- phone - телефон;
- is_seller - признак поставщика или исполнителя;
- is_buyer - признак покупателя.

Таблица materials хранит материалы:
- id - первичный ключ;
- code - код материала;
- name - наименование материала;
- unit - единица измерения;
- price - цена материала.

Таблица products хранит готовую продукцию:
- id - первичный ключ;
- code - код продукции;
- name - наименование продукции;
- unit - единица измерения;
- price - цена продажи.

Таблица specifications хранит спецификацию продукции:
- product_id - ссылка на продукцию;
- material_id - ссылка на материал;
- quantity - количество материала на одну единицу продукции.

Таблица orders хранит заказы покупателей:
- id - первичный ключ;
- order_number - номер заказа;
- order_date - дата заказа;
- executor_id - исполнитель заказа;
- customer_id - покупатель;
- total_amount - общая сумма заказа.

Таблица order_items хранит строки заказа:
- order_id - ссылка на заказ;
- product_id - ссылка на продукцию;
- quantity - количество продукции;
- unit - единица измерения;
- price - цена за единицу;
- subtotal - сумма строки заказа.

Таблица users используется приложением модуля 4:
- login - логин пользователя;
- password_hash - хэш пароля;
- role - роль пользователя;
- is_blocked - признак блокировки;
- failed_attempts - количество неудачных попыток входа.


4. СВЯЗИ МЕЖДУ ТАБЛИЦАМИ

customers.id связан с orders.executor_id - исполнитель заказа.
customers.id связан с orders.customer_id - покупатель заказа.
orders.id связан с order_items.order_id - один заказ содержит несколько строк.
products.id связан с order_items.product_id - строка заказа указывает на продукцию.
products.id связан с specifications.product_id - продукция имеет спецификацию.
materials.id связан с specifications.material_id - спецификация содержит материалы.

Связь между products и materials реализована через таблицу specifications, так как один продукт может состоять из нескольких материалов, а один материал может использоваться в разных продуктах.


5. РАСЧЕТ МАТЕРИАЛЬНОЙ СТОИМОСТИ

В модуле 3 рассчитывается полная материальная стоимость заказа покупателя. Для этого сначала определяется материальная стоимость одной единицы продукции по спецификации, затем полученная стоимость умножается на количество продукции в заказе.

Спецификация для пиццы "Пеперони" 33см.:

Куриное яйцо: 2 * 8.00 = 16.00
Майонез: 0.03 * 150.00 = 4.50
Бекон: 0.2 * 250.00 = 50.00
Тесто: 0.5 * 200.00 = 100.00
Сыр чеддер: 0.15 * 860.00 = 129.00
Сыр пармезан: 0.1 * 890.00 = 89.00
Томаты черри: 0.2 * 400.00 = 80.00

Итого материальная стоимость одной пиццы "Пеперони" 33см.:
16.00 + 4.50 + 50.00 + 100.00 + 129.00 + 89.00 + 80.00 = 468.50

В заказе указано 4 пиццы "Пеперони" 33см.

Полная материальная стоимость по заказу:
4 * 468.50 = 1874.00

Контрольный результат SQL-запроса:
order_sale_total = 7660.00
order_material_total = 1874.00


6. ОПИСАНИЕ SQL-ЗАПРОСА МОДУЛЯ 3

Запрос использует обобщенные табличные выражения WITH.

В блоке product_material_cost рассчитывается материальная стоимость одной единицы продукции. Для этого таблица specifications соединяется с таблицей materials, после чего выполняется суммирование quantity * price по каждой продукции.

В блоке order_material_cost данные заказов соединяются со строками заказа, продукцией, покупателем и рассчитанной себестоимостью продукции. Для каждой строки заказа определяется материальная стоимость строки.

Итоговый SELECT группирует данные по заказу и выводит:
- идентификатор заказа;
- номер заказа;
- дату заказа;
- покупателя;
- товарную сумму заказа;
- полную материальную стоимость заказа.


7. ОПИСАНИЕ ПРИЛОЖЕНИЯ МОДУЛЯ 4

Модуль 4 реализован на Python с использованием стандартной библиотеки Tkinter. Внешняя библиотека Pillow не используется, поэтому приложение может запускаться на RedOS без установки дополнительных Python-пакетов.

Основные возможности приложения:
- авторизация пользователя по логину и паролю;
- проверка капчи-пазла;
- блокировка пользователя после трех неудачных попыток входа;
- разделение ролей admin и user;
- панель администратора;
- добавление новых пользователей;
- изменение роли пользователя;
- разблокировка пользователей.

Начальная учетная запись администратора:
логин: admin
пароль: admin

Пароль для новых пользователей по умолчанию:
12345


8. ОСНОВНЫЕ КЛАССЫ И МЕТОДЫ ПРИЛОЖЕНИЯ

UserManager отвечает за работу с пользователями.

UserManager.connect() выполняет подключение к PostgreSQL.
UserManager.init_db() создает таблицу users при необходимости и добавляет администратора admin/admin.
UserManager.authenticate(login, password, captcha_ok) проверяет логин, пароль, капчу и состояние блокировки.
UserManager.list_users() возвращает список пользователей для панели администратора.
UserManager.add_user(login) добавляет нового пользователя с паролем 12345.
UserManager.set_role(login, role) изменяет роль пользователя.
UserManager.unblock(login) снимает блокировку и сбрасывает счетчик ошибок.

CaptchaDialog отвечает за капчу.

CaptchaDialog._load_images() загружает изображения из папки images.
CaptchaDialog.swap_fragment(index) меняет местами выбранные фрагменты.
CaptchaDialog.check() проверяет, собрана ли капча правильно.

LoginWindow отвечает за окно входа.

LoginWindow.open_captcha() проверяет заполнение логина и пароля и открывает капчу.
LoginWindow.finish_login() завершает авторизацию и открывает нужное окно по роли пользователя.

AdminWindow отвечает за панель администратора.

AdminWindow.reload_users() обновляет таблицу пользователей.
AdminWindow.add_user() добавляет пользователя.
AdminWindow.change_role() изменяет роль.
AdminWindow.unblock_user() разблокирует пользователя.


9. ПОРЯДОК ЗАПУСКА ПРОЕКТА

Сначала необходимо подготовить базу данных.

В pgAdmin создать базу данных:
V4

Затем открыть Query Tool именно в базе V4 и выполнить файл:
01_exam_v4_all_queries.sql

После выполнения должны быть созданы таблицы:
customers
materials
products
specifications
orders
order_items
users

Для запуска приложения на RedOS нужно перейти в папку модуля 4:

cd M4_Приложение
chmod +x run_redos.sh
./run_redos.sh

Если запуск выполняется вручную:

python3 check_redos.py
python3 main.py


10. ПРОВЕРКА РАБОТОСПОСОБНОСТИ

После запуска приложения необходимо выполнить вход под администратором:

login: admin
password: admin

Затем нужно пройти капчу. После успешного входа должна открыться панель администратора со списком пользователей.

Для проверки работы панели администратора можно:
- добавить пользователя test;
- убедиться, что пользователь появился в списке;
- изменить его роль;
- выполнить несколько неправильных входов и проверить блокировку;
- разблокировать пользователя через панель администратора.

Если PostgreSQL недоступен, приложение может открыть интерфейс в локальном резервном режиме. Для демонстрации интерфейса это допустимо, однако для проверки сохранения пользователей в базе данных PostgreSQL должна быть запущена, база V4 должна существовать, а SQL-скрипт должен быть выполнен.
